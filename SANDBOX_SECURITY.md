# Execution Sandbox Hardening System

## Overview

The Execution Sandbox Hardening system provides secure, isolated execution environments for running untrusted code generated by AgentForge agents. This system implements multiple layers of security to prevent malicious code from compromising the host system.

## Security Layers

### 1. Containerization
- Uses Docker containers to isolate execution environments
- Each execution runs in a fresh container with no persistent state
- Containers are automatically cleaned up after execution

### 2. Resource Limits
- **CPU**: Configurable CPU cores and throttling
- **Memory**: Hard memory limits to prevent OOM attacks
- **Processes**: Maximum number of processes limit
- **Disk**: Storage limits to prevent disk exhaustion

### 3. System Call Filtering
- Uses seccomp-BPF to restrict system calls
- Only allows essential syscalls needed for testing
- Blocks dangerous operations like:
  - Module loading/unloading (`init_module`, `delete_module`, `finit_module`)
  - Raw socket operations
  - Privileged operations (`ptrace`, `mount`, `umount2`)
  - System administration (`reboot`, `kexec_load`, `init_module`)
  - Hardware access (`ioperm`, `iopl`)

### 4. Network Policies
- **None**: Complete network isolation (no inbound/outbound)
- **Internal**: Loopback-only network
- **Restricted**: Limited external access for package repositories
- **Full**: Unrestricted network (not recommended for untrusted code)

### 5. File System Isolation
- Read-only root filesystem by default
- Temporary filesystem for `/tmp` with size limits
- Bind-mounted source code as read-only where possible
- Private `/tmp`, `/dev/shm` to prevent interference

### 6. Privilege Restrictions
- Runs as non-root user (UID/GID 1000 by default)
- Drops all Linux capabilities by default
- No new privileges flag to prevent privilege escalation
- Restricts ability to gain additional capabilities

### 7. Namespace Isolation
- PID namespace isolation (separate process tree)
- IPC namespace isolation (System V IPC, POSIX message queues)
- UTS namespace isolation (hostname and domain name)
- Mount namespace isolation (private filesystem view)

## Configuration

The system uses `SandboxConfig` objects to define security policies:

```python
from app.services.sandbox_executor import SandboxConfig, SecurityLevel, NetworkPolicy, ResourceLimits

# Maximum security configuration
secure_config = SandboxConfig(
    security_level=SecurityLevel.MAXIMUM,
    network_policy=NetworkPolicy.NONE,
    resource_limits=ResourceLimits(
        cpu_cpus=0.5,      # Half CPU core
        memory_mb=256,     # 256MB RAM
        max_processes=5,   # Max 5 processes
    ),
    max_execution_time=30, # 30 second timeout
)

# Less restrictive configuration for trusted code
trusted_config = SandboxConfig(
    security_level=SecurityLevel.MINIMAL,
    network_policy=NetworkPolicy.FULL,
    resource_limits=ResourceLimits(
        cpu_cpus=2.0,
        memory_mb=1024,
    ),
    read_only_root=False,  # Need write access for some operations
)
```

## Usage

The sandbox system is integrated into the TestExecutor service, which is used by the Tester agent to validate generated code:

```python
from app.services.test_executor import test_executor

# Execute Python tests in a secure sandbox
result = await test_executor.execute_python_tests(
    source_files={"main.py": "print('Hello World')"},
    test_files={"test_main.py": "def test_hello(): assert True"}
)

print(f"Tests passed: {result.passed}")
print(f"Tests failed: {result.failed}")
print(f"Status: {result.status}")
```

## Security Considerations

### Defense in Depth
The sandbox implements multiple overlapping security mechanisms:
- If container escape fails, resource limits prevent DoS
- If process limit bypass occurs, seccomp blocks harmful syscalls
- If network escape attempt made, network policies restrict access
- If filesystem breakout happens, read-only mounts prevent modification

### Fail-Safe Defaults
- All security features enabled by default
- Principle of least privilege applied throughout
- Explicit denial for unspecified permissions
- Conservative resource limits

### Resource Protection
- Memory leaks contained within container limits
- CPU bursts throttled via CFS quota
- Process explosion prevented by PID limits
- Disk filling prevented by quota and cleanup

## Performance Considerations

### Startup Overhead
- Container startup adds ~100-500ms latency
- Image caching reduces repeated pull delays
- Filesystem preparation adds minimal overhead

### Resource Efficiency
- Containers share host kernel (no VM overhead)
- Memory limits prevent host exhaustion
- CPU limits ensure fair sharing
- Automatic cleanup prevents resource leaks

## Customization

### Custom Images
Organizations can create custom base images with:
- Pre-installed dependencies
- Security patches
- Custom monitoring agents
- Organizational certificates

### Profile Tuning
Adjust security levels based on trust:
- **MAXIMUM**: Untrusted user-generated code
- **STANDARD**: Semi-trusted internal code
- **MINIMAL**: Trusted infrastructure code

### Escape Hatches
For legitimate needs requiring additional privileges:
- Specific capability grants (`--cap-add`)
- Controlled network access (`--network`)
- Temporary filesystem adjustments
- Extended timeouts for complex operations
