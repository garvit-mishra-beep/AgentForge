# Glossary — AgentForge

---

### Agent
A single AI model instance assigned to a specific role within a team. Each agent has a role, a model assignment, and a system prompt. Agents communicate through structured handoffs, not peer-to-peer messaging. *See: AGENT_ROLES.md, agents table in SCHEMA.md*

### Acceptance Criteria
A list of specific, testable conditions that a task output must satisfy. Defined by the Team Lead during the planning phase and used by the Reviewer, QA Engineer, and Team Lead (delivery) to verify the output. *See: AGENT_ROLES.md#team-lead*

### Artifact
Any file generated by an agent during task execution. Artifacts include source code files, test files, configuration files, Dockerfiles, and CI workflow files. Stored in the `outputs` table. *See: SCHEMA.md#outputs*

### Agent Memory
Persistence of task outputs and learnings across task runs. Short-term memory is the LangGraph `AgentState` (duration: one task). Long-term memory is stored as embeddings in pgvector (duration: 180 days). *See: AGENT_MEMORY.md*

### Agent Graph
The LangGraph `StateGraph` that defines the agent workflow. Nodes are agent steps, edges are handoffs. The graph supports conditional routing (e.g., "if review fails → route back to backend"). *See: TECH_SPEC.md#langgraph-agent-graph*

### Escalation
When an agent cannot complete its step (after exhausting retries), the task is escalated to the human. The human can approve, provide guidance and retry, or cancel. *See: HALLUCINATION_GUARD.md#layer-5-human-escalation*

### Handoff
The structured transfer of data from one agent node to the next in the LangGraph. Handoffs include the producing agent's output, the original acceptance criteria, and any accumulated context. *See: AGENT_ROLES.md#handoff-protocol*

### Human-in-the-Loop (HITL)
A design pattern where the agent workflow pauses at defined gates to wait for human approval. Implemented via LangGraph interrupts. The human can approve, reject with feedback, or request revision. *See: TECH_SPEC.md#human-in-the-loop*

### Model Registry
A registry of all supported AI models with their capabilities (context window, cost, max output tokens, supported roles). Used to validate role-to-model assignments and for fallback chains. *See: MODEL_REGISTRY.md*

### Node (LangGraph)
A single step in the agent workflow graph. Each node is an async function that receives the current `AgentState` and returns an updated `AgentState`. Nodes correspond to agent roles (team_lead_plan, backend_implement, review, qa, security). *See: TECH_SPEC.md#langgraph-agent-graph*

### Orchestrator
The component that creates the LangGraph StateGraph, initializes the state from a task, invokes the graph, and handles errors and interruptions. Located in `apps/api/agents/orchestrator.py`.

### Output Panel
The frontend UI panel that displays agent-generated output. Shows streaming token chunks during execution, and final file contents with syntax-highlighted diff view after completion. *See: apps/web/components/agents/OutputPanel.tsx*

### Prompt Template
A Jinja2 template file containing the system prompt for an agent role. Templates include injectable variables (`{task}`, `{codebase_summary}`, `{acceptance_criteria}`). Versioned (v1.0, v1.1, etc.). *See: AGENT_PROMPTS.md*

### Review Gate
A workflow checkpoint where one agent's output is reviewed by another agent (the Reviewer) before proceeding. Blocking issues cause a revision cycle; non-blocking issues are logged and passed through. *See: TECH_SPEC.md#data-flow*

### Role
A defined job function within an AgentForge team. The 6 roles are: Team Lead, Backend Engineer, Frontend Engineer, QA Engineer, Security Engineer, DevOps Engineer. Each role has a system prompt, expected output format, and handoff protocol. *See: AGENT_ROLES.md*

### Supervisor Node
A conceptual node in the LangGraph that reads the current state and routes execution to the next appropriate agent node. In practice, this is implemented via LangGraph's conditional edges. *See: TECH_SPEC.md*

### Task
A unit of work assigned to an AgentForge team. Contains a title, description, and optional acceptance criteria. Executed by the LangGraph workflow, producing steps, messages, and outputs. *See: SCHEMA.md#tasks*

### Task Step
A single execution step within a task. Each step corresponds to one agent node invocation. Tracks: agent, role, model, tokens used, cost, confidence, input, output, timestamps. *See: SCHEMA.md#task_steps*

### Team
A group of AI agents (each with a role and model assignment) that work together on tasks. Belongs to a project. Multiple teams can exist per project for different purposes (e.g., "Backend Team", "Full Stack Team"). *See: SCHEMA.md#teams*

### Workflow
The complete sequence of agent steps executed for a task. Defined by the LangGraph graph structure. Can include branching, conditional routing, revision cycles, and human-in-the-loop interrupts. *See: TECH_SPEC.md*

### Workspace
The shared context for a task execution. Includes: codebase summary, file contents, agent outputs, acceptance criteria, and accumulated state. The workspace is the LangGraph `AgentState`.
